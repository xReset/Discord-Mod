# DiscordMod — Custom Discord Client Mod

**Location (code):** `E:\DiscordMod`
**Target:** Discord **Stable**, Windows 11
**Stack:** JavaScript / Electron (NOT Python — fully separate from the `E:\Selfbot` Python selfbot)
**Status:** working foundation + first feature (deleted-message viewer) functional as of 2026-05-29.

---

## What this is / why it exists

A self-built Discord client mod — our own injector, **not** Vencord / BetterDiscord / any
closed-source client. We own every line. It loads the official Discord app untouched and adds
our own code that runs *inside* the client.

### The problem it solves
The `E:\Selfbot` Python selfbot is **headless and server-side** — it only sees messages *after*
Discord's servers broadcast them (via the gateway `on_message`). It can never:
- transform a message *before* it sends (only edit-after-send, which leaves an "(edited)" tag / flash),
- touch the local client UI,
- reveal info the client hides from the user.

A client mod runs **inside the Electron client**, on the path between the user's keyboard/eyes and
Discord's servers. That unlocks an entirely different feature axis the selfbot fundamentally cannot reach.

### The split (selfbot vs client mod)
| | Selfbot (`E:\Selfbot`, Python) | Client mod (`E:\DiscordMod`, JS) |
|---|---|---|
| Runs | headless, 24/7, even when client closed | only while Discord is open on this PC |
| Acts on | Discord's data (everyone sees) | local view + the user's own input |
| Best at | automation, background tasks, reacting to others | UI, pre-send transforms, revealing hidden info |
| Can't | touch client UI / pre-send | run when client closed, background automation |

They are **complementary**, not overlapping.

## First feature (the "test"): Deleted-message viewer

Goal: when any message is deleted in a channel the user is viewing, it should **stay visible,
painted red with a `(deleted)` tag** instead of vanishing — so the user always sees what was deleted,
clearly signalled (not a vanilla message).

**How it works:** the renderer grabs Discord's internal Flux dispatcher and adds an *interceptor*.
On `MESSAGE_DELETE` / `MESSAGE_DELETE_BULK` it records the message id and **returns `true` to block
the removal**, so Discord never takes the message out of the store/DOM. A CSS class + a
`MutationObserver` then paint the message content red and re-apply the styling when Discord
re-renders / virtualizes the list on scroll.

Console controls (DevTools console): `DCMod.toggleDeleted()`, `DCMod.clearDeleted()`.

## Architecture

```
app.asar (OUR shim — generated by install.js, packed)
  ├─ index.js    main process: patches BrowserWindow, then require()s the original app
  │              - sets webPreferences.devTools = true  (Stable disables it)
  │              - binds Ctrl+Shift+I via before-input-event
  │              - swaps webPreferences.preload -> our preload (stashes original in env)
  └─ preload.js  - require()s Discord's ORIGINAL preload first (keeps DiscordNative etc.)
                 - injects renderer.js into the page MAIN WORLD via
                   webFrame.executeJavaScript()  (NOT a <script> tag — see CSP note)

_app.asar        untouched backup of the original Discord app (created on first install)

src/renderer/renderer.js   the actual feature code; main-world script.
                           Read fresh from disk every launch -> edit + restart = iterate,
                           NO repack needed.
```

### Why these specific choices (hard-won)
- **`webFrame.executeJavaScript` instead of injecting a `<script>` element** — Discord's
  Content-Security-Policy blocks inline `<script>` (`Refused to execute inline script...`).
  `executeJavaScript` runs at the Electron level in the page's main world and bypasses CSP.
- **DevTools enabled in the shim, not via `settings.json`** — Discord Stable rewrites
  `%APPDATA%\discord\settings.json` from memory on exit and wipes the
  `DANGEROUS_ENABLE_DEVTOOLS_...` flag. Forcing `webPreferences.devTools = true` at window
  creation is permanent and can't be wiped.
- **Webpack chunk push happens AT MOST ~6 times, only after webpack has booted
  (`chunk.length >= 5`)** — CRITICAL. Spamming `webpackChunkdiscord_app.push(...)` on a tight
  retry loop **during the login / remote-auth phase logged the account out** (corrupts the auth
  handshake). The boot loop now polls patiently (500ms, up to ~5min) and only pushes once the app
  is past login.

## Install / use

> Requires Node.js. **Fully quit Discord first** (tray → Quit, or `Stop-Process -Name Discord -Force`)
> or the .asar files are locked.

```powershell
cd E:\DiscordMod
npm install            # @electron/asar (dev dep, used by installer)
node install.js
```

Relaunch Discord → `Ctrl+Shift+I` → Console should show:
```
[DCMod] dispatcher hook installed — deleted-message viewer active
[DCMod] ready ✓
```

### Iterate on features
Edit `src/renderer/renderer.js`, then **fully restart Discord**. No re-install — preload reads the
file fresh each launch.

### After a Discord update
Updates create a new `app-<version>` folder with a fresh untouched `app.asar`. Re-run
`node install.js` to re-inject.

### Uninstall
Quit Discord, `node uninstall.js` (restores original `app.asar` from `_app.asar`), restart.

## Known caveats / risk
- **ToS:** client modding violates Discord ToS — same risk bucket as the existing selfbot.
- **Maintenance:** patches rely on undocumented internals (Flux dispatcher shape, webpack,
  `message-content-<id>` DOM ids). Discord updates can break them; that's the main ongoing cost.
- **Blocking `MESSAGE_DELETE`** is local-only and may skip some unread/ack side effects — fine for
  a personal viewer.
- Multi-window preload race: original preload path is stashed in `process.env` — fine for
  sequential window creation (normal personal use).
